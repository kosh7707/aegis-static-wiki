# S4. SAST Runner 인수인계서

> **반드시 `docs/AEGIS.md`를 먼저 읽을 것.** 프로젝트 공통 제약 사항, 역할 정의, 소유권이 그 문서에 있다.

## 1. 역할과 경계

### 너는

- **SAST Runner 전담 개발자** (`services/sast-runner/`)
- `wiki/canon/api/sast-runner-api.md` API 계약서 소유
- `wiki/canon/specs/sast-runner.md` 명세서 소유
- `scripts/start-sast-runner.sh` + `services/sast-runner/.env` 소유
- 12개 엔드포인트 관리: scan (동기+NDJSON 스트리밍+durable ownership), functions, includes, metadata, libraries, build, build-and-analyze, discover-targets, health, request status, request result, request cancel
- Durable ownership error contract — `REQUEST_NOT_FOUND`, `REQUEST_EXPIRED`, and `REQUEST_ID_CONFLICT` public transport errors preserve legacy fields while carrying `success=false` and standard `errorDetail{code,message,requestId,retryable=false}`.
- build path는 execution-only. SDK/toolchain/build-command 해석은 하지 않음
- `metadata.cweId` 표준화 — 전 도구에서 CWE 식별자를 `cweId` 필드로 제공 (S2 Finding 매핑용)
- `metadata.evidenceResolution` + enriched `scan.sca.libraries[]` — SAST/SCA evidence 해상도를 높이는 deterministic projection (보안 판정/CVE 조회 아님)
- `staticEvidenceContract` v1 구현 — `wiki/canon/specs/sast-runner-static-evidence-contract.md`가 canonical Coverage/Readiness/Claim-boundary/toolEvidenceMatrix contract이며, `/v1/scan` 및 `/v1/build-and-analyze`에 additive로 부착된다. Per-tool anomaly는 `systemStability=degraded`와 `coverage.staticToolExecution=partial/anomalyReasonCodes[]`로 전파된다.
- Golden Corpus v1 / Tool Portfolio Governance v1 — `services/sast-runner/tests/fixtures/golden_corpus_v1/manifest.json`, `benchmark/static_evidence_report.py`, `benchmark/tool_portfolio_governance.py`, `wiki/canon/specs/sast-runner-tool-portfolio-governance-v1.md`가 validation/report/governance baseline이다. 현재 decision은 `keep-current-six-tools`다. Golden Corpus v1은 structural graph, SCA diff partial, degraded execution, policy failure, CWE-120, CWE-190, CWE-416 canary를 포함한다.
- Corpus Readiness Gate v1 — `benchmark/tool_portfolio_corpus_readiness.py`가 required external corpora의 local path/file/checksum/split readiness를 검증한다. 현재 S4 harness fixture는 `juliet-c-cpp-1.3`을 required corpus로 요구하며, local Juliet corpus 부재 때문에 `corpusReadinessGate.status="blocked"` / `LOCAL_JULIET_CORPUS_NOT_PRESENT` / `decisionGradeReady=false` 상태가 정상이다. `python -m benchmark.tool_portfolio_corpus_readiness` CLI로 독립 preflight 실행이 가능하며, required corpus 미선언은 `CORPUS_REQUIRED_CORPORA_NOT_DECLARED`로 fail-closed된다. Required corpus IDs are strict no-whitespace/no-control safe strings; malformed IDs fail closed with `CORPUS_REQUIRED_CORPUS_ID_INVALID` and are not echoed. Safe unknown required corpora project as generic `external`. Report composition에서는 `corpusReadinessGate`가 authoritative라서 legacy `external_corpus_status`가 available이어도 readiness `not_run`/`blocked`를 덮어쓸 수 없고, caller-provided readiness `externalCorpusStatus`는 canonical keys/allowed nested fields/allowlisted readiness reason codes로 sanitized된다. Corpus manifest/report strict paths reject blank/absolute/Windows-drive-or-UNC/backslash-normalized traversal `sourcePath` values without raw echo; readiness intentionally returns blocked JSON with `CORPUS_CASE_SOURCE_PATH_UNSAFE` for the same normalized unsafe paths instead of treating them as available/missing files, and unsafe case statuses redact raw paths as `sourcePath="<unsafe>"` plus `sourcePathStatus="unsafe"`; CLI invalid-output JSON uses fixed sanitized error text and safe `errorClass` instead of raw exceptions/paths; output-write failures fall back to sanitized stdout JSON with `CORPUS_READINESS_OUTPUT_WRITE_FAILED`; case/acquisition statuses no longer expose host-local `resolvedPath`, `localPath`, or `resolvedLocalPath`, using status-only fields instead; absent acquisitions use `localPathStatus="not_declared"` and `resolvedLocalPathStatus="not_resolved"`. `benchmark/tool_portfolio_corpus_acquisition.py`는 실제 NIST Juliet/SARD archive를 다운로드·SHA-256 검증·safe extract한 뒤 `.omx/corpora/s4-tool-portfolio/`에 acquisition/corpus/readiness manifest를 만든다. 2026-05-13 실제 Juliet/SARD cache는 80 focused cases로 readiness `available`이다.
- Experiment report forbidden-key guard error-surface sanitization — final offline report `_reject_forbidden_keys()` canonicalizes fixed forbidden verdict literals and redacts arbitrary mapping path labels/object stringification; it is a last-defense invariant, so valid report generation should stay clean while failure diagnostics remain value-free.
- Corpus manifest identity-field error-surface sanitization — `caseId`, `expected.targetId`, explicit `lineageId`, and external `acquisitionId` must be safe identifiers (`[A-Za-z0-9][A-Za-z0-9._:-]{0,255}`, no whitespace/control/surrounding whitespace). Duplicate IDs, lineage/source-artifact leakage, acquisition lookup misses, and acquisition checksum mismatches use value-free diagnostics, and absent public lineage falls back to safe `caseId`, never raw `sourceRef`.
- Corpus manifest forbidden-key error-surface sanitization — recursive forbidden verdict-key validation reports only fixed forbidden key literals and sanitized path components; arbitrary parent labels and `str`/`repr` of key objects are not echoed.
- Corpus manifest case checksum strictness — `s4-tool-portfolio-experiment-corpus-v1` case `checksum` is exact `sha256:<64 lowercase hex>`. Prefix-valid malformed values are manifest-invalid and cannot become readiness `expectedChecksum` evidence.
- Corpus Readiness report-side input sanitization — caller-provided offline `corpusReadinessGate` now has a strict top-level allowlist, forbidden caller `inputValidation`, exact optional schema/policy validation, field-aware nested `summary`/`acquisitionStatuses`/`caseStatuses` validation, generated `requiredCorpusInputValidation` compatibility, and generated `consistencyChecks` re-ingest compatibility. Normalization reconstructs only validated fields and must not raw-preserve `{**gate}`.
- Actual runner CLI input failure handling — JSON input-loading failures and argparse scalar failures for `--timeout`/`--phase` return exit `1` with fixed stderr `input validation failed`; invalid scalar inputs fail before file reads/output writes, valid positive decimal timeouts preserve execution selection, and downstream build/report exceptions remain outside this catch.
- Actual runner path-boundary diagnostics — source-root escape and relative acquisition-base escape failures report only stable category messages and suppress underlying `Path.relative_to()` exception chains, preventing host-local paths from surfacing through tracebacks/log aggregation.
- Juliet corpus file selection diagnostics — missing Juliet source file selection failures report only `No Juliet file found` and never echo host-local corpus/cache roots.
- Actual runner JSON loader diagnostics — read, parse, and non-object JSON input failures are value-free and suppress host-local paths, content fragments, parser internals, and exception chains.
- Corpus Readiness CLI/JSON loader diagnostics — parser/arg-shape failures emit sanitized invalid-input JSON on stdout with exit `1`, no raw argparse stderr/usage/token echo, and no JSON load/readiness-build side effects. Non-object JSON helper failures report only the fixed object-required category and never echo host-local input paths; CLI invalid-output sanitization remains the public surface.
- Tool-set config validator diagnostics — `validate_tool_set_config()` keeps invalid current/future/unknown/non-string config errors value-free while preserving valid current-six and explicit `allow_future=True` passthrough behavior.
- Decision-cycle forbidden runtime coupling guard — `assert_no_forbidden_runtime_coupling()` still enforces no-network/no-LLM/no-S5 imports/calls, but failure diagnostics are value-free and do not echo host-local paths or regex internals.
- Corpus acquisition helper/CLI diagnostics — zip member safety, outside-cache deletion guard, and archive checksum mismatch errors use value-free categories only; member names, host-local paths, expected checksums, and actual checksums are not echoed. The acquisition CLI also maps invalid `--corpus`, unknown arguments, and malformed option shapes to fixed `input validation failed` before acquisition/network/summary side effects, while valid repeated corpus selection preserves caller order. Its summary-output and stdout JSON emission boundaries now use fixed `corpus acquisition output failed` with best-effort stderr fallback and no raw exception/object/path echo.
- Acquisition manifest duplicate-index diagnostics — `build_acquisition_index()` keeps duplicate `acquisitionId` failures value-free (`duplicate acquisitionId`) so safe-shaped local corpus IDs are not echoed in error surfaces.
- Acquisition manifest real-calendar parser diagnostics — `downloadedAt` format-matching but impossible calendar values fail with value-free field-only diagnostics and suppress the underlying `datetime.strptime` exception chain, so tracebacks/logging do not expose parser internals or raw parse context.
- Corpus Acquisition provenance/split/schema hardening — acquisition manifest는 required fields와 known optional fields(`sourcePageUrl`, `expectedArchiveChecksum`)만 허용하고, exported checksum/validation이 같은 canonicalizer를 사용한다. Unknown/non-JSON fields는 checksum 계산 전에 raw value leakage 없이 fail-closed된다. `downloadedAt`은 exact date/UTC timestamp + real calendar로 검증되고, `sourceUrl`은 `http`/`https`/`file`/`local`, `sourcePageUrl`은 `http`/`https`만 허용한다. Existing extraction cache는 prior acquisition manifest가 verified archive checksum, recomputed tree checksum, local path, `expectedArchiveChecksum=sha256:<hex>` source metadata와 일치할 때만 재사용한다. Metadata-only `sourcePageUrl`/`licenseOrRedistributionNote` 변경은 재추출 없이 manifest만 re-pin하고 original `downloadedAt`을 보존한다. Single SARD candidate는 validation/test 양쪽에 복제하지 않는다. Corpus manifest `createdAt`은 pinned acquisition manifest 날짜에서 파생되어 trusted cache 재실행 시 checksum drift를 만들지 않으며, validator도 sourceRef/checksum이 같은 external source artifact가 validation/test 양쪽에 등장하면 split leakage로 거부한다.
- Actual Tool Portfolio runner — `benchmark/tool_portfolio_actual_runner.py`는 pinned local Juliet/SARD corpus manifest를 case-only staging tree로 복사한 뒤 current-six portfolio configs를 실행해 offline `s4-tool-portfolio-experiment-report-v1`을 만든다. Staging은 `semgrep`/`cppcheck`/`flawfinder`가 full extracted cache를 스캔하지 못하게 하는 필수 격리층이다. CLI supports `--base-path` for relative acquisition `localPath`; direct staging rejects cwd-dependent relative `localPath` without explicit base, while report mode returns readiness-blocked/no-scan with `LOCAL_CORPUS_BASE_PATH_REQUIRED`; top-level system gate uses only `full-current-six` executions, while single/leave-one-out comparative completeness is separate. 2026-05-13 actual low-threshold run initially exposed fake Juliet SDK lookup and Flawfinder malformed numeric parsing; both are fixed. `report-after-stage-path-fix.json` is runner-integrity/diagnostic evidence only and now has final `qualityGate.status="not_decision_grade"`, not quality sufficiency, deterministic `toolContributionDiagnostics`, and absolute staged acquisition paths even with relative CLI `--work-dir`.
- Static Evidence consumer canaries — `benchmark/static_evidence_consumer_canary.py`와 `tests/fixtures/static_evidence_contract/consumer_canaries/*.json`은 S3-facing JSON contract 소비 semantics를 S4 내부 app import 없이 검증한다. Summary projection은 field-specific allowlist로 gate statuses/generated reasons/surfaces/claim IDs/tool IDs/policies만 내보내며, unsafe projection 또는 caller-spoofed summary-only diagnostic은 canary-generated `STATIC_EVIDENCE_CONTRACT_UNSAFE_PROJECTION` + `localStaticEvidenceReady=false`로 fail-closed된다.
- Tool Portfolio report consumer canary — `benchmark/tool_portfolio_report_consumer_canary.py`는 offline `s4-tool-portfolio-experiment-report-v1`만 입력으로 받는 pure JSON helper/CLI smoke gate다. `toolPortfolioDecisionGradeUsable`은 offline Tool Portfolio quality evidence 사용 가능성만 뜻하며 security verdict, routing, S3/S5 decision이 아니다. Helper는 system/corpus/local-quality/threshold/final gate가 모두 pass-like이고, diagnostic surfaces가 available이며, current-six tool contribution rows가 모두 있고, sanitized failure reason/follow-up이 비어 있을 때만 true를 낸다. Diagnostic candidate IDs are advisory, not required for clean reports. reason/follow-up/status/intent/diagnostic/tool identifier는 allowlist로 sanitize한다. Summary itself is versioned as `s4-tool-portfolio-report-consumer-summary-v1`, and tests lock the exact top-level summary key set. Forbidden decision/routing/verdict vocabulary나 raw diagnostic/tool result detail은 exact key/value canary로 차단된다. CLI smoke: `.venv/bin/python -m benchmark.tool_portfolio_report_consumer_canary --report <path> [--require-decision-grade]` emits sanitized summary JSON; exit `0` means parsed valid report summary OK and any optional decision-grade requirement passed, `2` means summary emitted but the report is parsed-invalid (`reportPresent=false`) or required decision-grade failed, `1` means no summary emitted due CLI/file/JSON syntax input failure. The committed sample `benchmark/results/tool_portfolio/s4-harness-fixture-consumer-summary-v1.json` and all other summary-schema artifacts are guarded: only that canonical path may carry `summarySchemaVersion="s4-tool-portfolio-report-consumer-summary-v1"`, and its parsed JSON must exact-match the generated canonical report summary; extra or drifted summary artifacts are blocked by S4 tests. Projected diagnostic identifiers and summary-only reason spoofing are also fail-closed: unknown/malformed `candidateId`, `toolId`, or `evidenceClass` values are omitted but add canary-generated `TOOL_PORTFOLIO_REPORT_UNSAFE_PROJECTION` and force non-decision-grade usability; caller-provided `TOOL_PORTFOLIO_REPORT_ABSENT` / `MALFORMED` / `SCHEMA_UNSUPPORTED` / `UNSAFE_PROJECTION` reason codes inside present valid reports are not copied as producer reasons. Malformed `reasonCodes` / `decisionSupport.requiredFollowUps` containers or list items also fail closed without stringifying malformed objects while preserving valid allowlisted entries from proper lists. Present non-string or blank scalar projections fail closed as unsafe projection, while absent/`None` scalar fields remain compatible. Positive gate predicates are therefore necessary but insufficient: otherwise-positive reports with missing diagnostics, incomplete current-six contribution rows, non-empty sanitized failure reasons, or non-empty required follow-ups are marked with canary-generated `TOOL_PORTFOLIO_REPORT_UNSAFE_PROJECTION` and `toolPortfolioDecisionGradeUsable=false`, while already non-decision-grade/failing reports do not gain unsafe solely from completeness gaps. `runnerIntegrityOnly` is likewise gated by unsafe projection: sanitized threshold/reason context remains visible, but the convenience boolean is false whenever `TOOL_PORTFOLIO_REPORT_UNSAFE_PROJECTION` is present.
- Claim Support Readiness — `app/scanner/claim_support_gate.py`와 `tests/fixtures/claim_support_gate_v1/manifest.json`은 Quality Gate가 concrete SAST tool identity가 아니라 normalized evidence + claim boundaries만으로 bounded claim support를 판정하도록 잠근다. Runtime `qualityEvaluation`은 여전히 `not_evaluated`다.
- Tool Output Compatibility v1 — `benchmark/tool_output_compat.py`와 `tests/fixtures/tool_output_compat_v1/manifest.json`은 현재 6개 도구 raw output parser 호환성을 외부 도구 실행 없이 잠근다. Governance `parserCompatibility` gate가 이 report를 소비한다.
- Tool Output Compatibility manifest/parserKind diagnostics — `tool_output_compat.load_manifest()` reports unreadable, malformed, and non-object manifests with stable value-free categories and suppressed read/parser exception chains; unsupported `parserKind` does not echo caller-controlled values.
- Tool Output Compatibility fixture input diagnostics — parser fixture paths are resolved as safe relative paths under the manifest directory; missing fixtures and malformed SARIF JSON emit stable value-free categories with suppressed read/parser causes, and traversal/absolute fixture paths are rejected without raw echo.
- Benchmark Slice Report v1 — `benchmark/benchmark_slice_report.py`는 pinned historical Juliet baseline JSON만 읽어 variant-01 precision/FP와 all-variant noise evidence를 source-scoped로 제공한다. Governance `benchmarkSliceCoverage` gate가 이 evidence를 소비한다.
- Benchmark Slice artifact loader diagnostics — `benchmark_slice_report._load_artifact()` reports unreadable, malformed, and non-object baseline artifacts with stable value-free categories and suppressed parser/read exception chains; baseline file paths and content fragments are not echoed.
- Juliet benchmark scan-failure diagnostics — `benchmark/juliet_runner.py::_benchmark_cwe()` preserves all-cases-FN fallback on scan failure but logs only stable CWE failure categories, not raw orchestrator/tool exception text.
- Juliet manifest discovery diagnostics — `discover_cwe_suites()` reports absent `testcases/` with stable `Juliet testcases directory not found` and does not echo host-local Juliet root paths.

### 너는 하지 않는다

- DGX Spark / LLM Engine 관리 -> **S7**
- CVE 조회 -> **S5** (`POST /v1/cve/batch-lookup`으로 이관 완료)
- 프롬프트 작성, LLM 응답 파싱 -> S3
- 지식 그래프, 벡터 검색 -> S5
- UI -> S1
- `scripts/start.sh` / `scripts/stop.sh` 직접 수정 금지 -> S2에 work-request

### Codex / OMX 운영 메모

- 하드 가드레일 재확인:
  - S4는 **다른 서비스 코드를 읽지 않는다**.
  - 다른 서비스와의 소통은 **WR로만** 한다.
  - 연동 판단은 `wiki/canon/api/` 계약서만 보고, 부족하면 담당자에게 WR을 보낸다.
  - **커밋은 하지 않는다**. 커밋은 S2 세션만 한다.
  - `scripts/start*.sh`, `scripts/stop*.sh`, 서비스 기동 명령은 **사용자 허락 없이 실행하지 않는다**.
  - 로그/장애 분석은 `log-analyzer` MCP를 우선 사용한다.
- lane 전용 작업 메모와 후속 세션 인계는 `wiki/canon/handoff/s4/`와 `.omx/state/sessions/{session-id}/...`를 우선 사용한다.
- 공용 `.omx/notepad.md`, `.omx/project-memory.json`에는 전역 durable 정보·공통 운영 규칙·cross-lane에 실제 필요한 사실만 짧게 남긴다.
- **`$ralph`**: 스캐너 안정화, 빌드 파이프라인 복구, 벤치마크 개선처럼 한 lane이 끝까지 파고들어야 하는 작업에 우선 사용한다.
- **`$team`**: S2 계약 조정, S5 CVE/KB 연동, S7 모델/프롬프트 영향 점검처럼 여러 lane이 동시에 얽히는 작업에 우선 사용한다.
- **`$trace`**: 이전 Codex/OMX 세션의 검증 흐름, 실패 원인, 반복 실험을 복기할 때 사용한다.
- skill을 써도 **소유권, API 계약, work-request 규칙은 변하지 않는다**.

현재 상태 해석:
- `policyStatus="ok"` / all alive는 **시스템 안정성** 신호다.
- offline Tool Portfolio `systemStabilityGate`는 required tool set이 비어 있으면 `SYSTEM_REQUIRED_TOOLS_NOT_DECLARED`로 fail-closed되고, unknown required tool은 `REQUIRED_TOOL_UNKNOWN` preflight failure로 기록한다.
- offline Tool Portfolio `systemStabilityGate.status="not_run"`은 local metric evidence를 보존하지만 final `qualityGate.status="pass"`를 만들 수 없고 `SYSTEM_STABILITY_GATE_NOT_RUN`으로 `not_decision_grade`에 머문다.
- offline Tool Portfolio `qualityGateAllowed=true`는 `systemStabilityGate.status="pass"`에만 허용되며, default/precomputed `not_run` gate는 `qualityGateAllowed=false`를 낸다. Caller-provided 모순 gate는 `SYSTEM_STABILITY_GATE_INCONSISTENT`로 fail/blocked 정규화된다.
- offline Tool Portfolio caller-provided `systemStabilityGate` payload가 non-mapping이면 `SYSTEM_STABILITY_GATE_INPUT_INVALID`로 fail-closed된다. `status="pass"`는 current-six `requiredTools` 완전성과 `preflight`/`executionCompleteness` nested pass evidence(`status="pass"`, empty `failures`)를 증명해야 하며, fail/not_run minimal evidence는 input-invalid로 덮어쓰지 않는다.
- local harness의 `qualityGate.localQualityAssessment.status="fail"`은 **품질 threshold** 신호다.
- local Quality Gate threshold의 `requiredSplits`가 명시적으로 비어 있거나 blank-only이면 `QUALITY_REQUIRED_SPLITS_NOT_DECLARED`로 fail-closed된다.
- `findings_by_config`가 required current-six config를 빠뜨리거나 value/element shape가 잘못되면 `FINDINGS_BY_CONFIG_INPUT_INVALID`로 fail-closed되고 scored-looking metrics를 만들지 않는다. Finding `toolId`도 scoring 전 검증된다: unknown/missing/blank tool IDs, `single-tool:<tool>` mismatch, and `leave-one-out:<tool>` excluded-tool contamination are fail-closed without raw unknown tool echo. Mapping finding payload도 scoring 전에 검증/정규화된다: valid top-level `file`/`line`은 canonical `location`으로 복사되고, malformed rule/location/file/line/metadata/dataFlow는 raw value echo 없이 `FINDINGS_BY_CONFIG_INPUT_INVALID`로 차단된다.
- oracle `matchingPolicy` payload가 mapping/object가 아니면 `ORACLE_MATCHING_POLICY_INPUT_INVALID`로 fail-closed되고 scored-looking metrics를 만들지 않는다.
- oracle `matchingPolicy` semantic fields는 strict하다: optional 누락은 canonical default(`lineWindowDefault=5`, `functionFallbackDefault=false`)로 정규화되고, schema drift/unknown key/bool·non-int·out-of-range line window/non-bool fallback은 `ORACLE_MATCHING_POLICY_INPUT_INVALID`로 oracle scoring 전 fail-closed된다.
- legacy `external_corpus_status`는 compatibility context일 뿐이며 reserved/readiness-owned key, forbidden verdict vocabulary, malformed reasonCodes/acquisitionIds, invalid status는 sanitized `legacyExternalCorpusStatusInputValidation`으로만 기록되고 final `qualityGate`/`requiredFollowUps`에는 영향을 주지 않는다.
- offline Tool Portfolio `runId`/`createdAt`/`phase`는 decision-cycle 생성 전에 검증된다. invalid identity는 safe placeholder로 정규화되고 `REPORT_IDENTITY_INPUT_INVALID`/`reportIdentityValidation`으로 fail-closed되며 raw secret string이 `runId`/`decisionCycleId`에 남지 않는다.
- offline Tool Portfolio `thresholds`는 decision-cycle checksum 전에 JSON-serializable 여부를 검사한다. invalid object/set payload는 sanitized checksum surrogate를 사용하고 `QUALITY_THRESHOLDS_INPUT_INVALID`로 local quality만 fail-closed되며, system/matching/findings가 유효하면 split metric scoring은 유지된다.
- offline actual-runner CLI input-loading and scalar parser failures are fail-closed: missing/malformed/non-object JSON inputs plus invalid `--timeout`/`--phase` argparse selections return exit `1` with fixed stderr and no traceback/path/content/usage/token echo; invalid scalar inputs do not read manifest files or write output artifacts.
- offline actual-runner path-boundary validation suppresses `Path.relative_to()` exception chains for source-root symlink escapes and relative localPath base escapes; direct diagnostics stay category-only.
- offline Benchmark Slice artifact loading is value-free on failure: missing/unreadable files, malformed JSON, and non-object JSON do not echo baseline paths/content and suppress chained read/parser exceptions.
- offline Tool Output Compatibility manifest loading and parser-kind selection are value-free on failure: missing/malformed/non-object manifest inputs and unsupported parser kinds do not echo paths, content, parser internals, or caller-controlled parser labels.
- offline Tool Output Compatibility fixture inputs are value-free on failure: unsafe `inputFixture` paths, missing files, and malformed Semgrep SARIF JSON do not echo paths, content, parser internals, or chained read/parser exceptions.
- Semgrep SARIF parser malformed-output failures are value-free: parser-shape errors and non-mapping wrappers map to stable `Failed to parse SARIF output` without parser internals, caller strings, or chained exceptions.

- Build metadata `compiler` evidence must remain identity-only (basename + optional version); never return resolved SDK/compiler absolute paths in `/v1/metadata` or nested build metadata.

- `files[].path` validation rejects drive-qualified relative forms: single-letter drive prefixes such as `C:foo.c` are blocked with `Absolute path not allowed`, while arbitrary colon filenames like `src:main.c` remain allowed.
- Semgrep runner raw stdout/stderr is not logged on empty-output or non-JSON-output paths; logs retain only fixed diagnostic categories and response/error semantics are unchanged.
- gcc-fanalyzer and scan-build per-file exception strings are not logged during partial aggregation; logs retain analyzer/file categories while failure counters and partial semantics remain intact.
- malformed scan-build plist parse failures do not log plist filenames or parser exception text; other valid plist files in the same output directory are still parsed.
- offline Juliet corpus file-selection failures are path-redacted; missing fixture diagnostics do not echo corpus/cache root paths.
- offline actual-runner JSON object loading is read/parse sanitized: missing files, malformed JSON, and non-object JSON do not echo host paths or input content.
- offline Corpus Readiness JSON object loading is path-redacted for non-object inputs; direct helper diagnostics and CLI invalid payloads do not echo host-local manifest paths.
- offline Tool Portfolio tool-set config validation is value-free for invalid values: unknown current-tool, future-WR-gated, unknown config, and non-string config diagnostics do not echo the submitted config/tool.
- offline decision-cycle coupling guard diagnostics are value-free: forbidden network/LLM/S5 coupling failures do not echo host-local paths or regex internals.
- offline corpus acquisition helper diagnostics are value-free for local path/checksum surfaces: unsafe zip member, outside-cache deletion, and checksum mismatch errors do not echo member names, host paths, expected checksums, or actual checksums.
- offline acquisition index duplicate detection is value-free: duplicate `acquisitionId` errors identify only the duplicate category and never echo the ID.
- offline acquisition manifest real-calendar validation suppresses parser exception chains: impossible date/timestamp values do not echo raw values and do not retain `datetime.strptime` as `__cause__`.
- offline acquisition manifest checksum은 strict canonical schema만 입력으로 받는다. Unknown fields, non-string keys, malformed optional provenance fields, and non-JSON values are rejected before serialization; `sourcePageUrl`/`expectedArchiveChecksum` remain accepted and included in the checksum.
- offline acquisition provenance semantics are validated locally: `downloadedAt` must be exact date/UTC timestamp with real calendar validity; `sourceUrl` keeps `local://` fixture compatibility while rejecting malformed schemes/hosts/paths; `sourcePageUrl` is http/https only.
- offline acquisition manifest error surfaces are sanitized: safe unknown field labels are preserved, unsafe keys are redacted as `<unsafe>`, and malformed http(s) authorities with hostless/query/fragment/userinfo forms are rejected without raw URL echo.
- local Quality Gate `thresholds` payload가 mapping/object가 아니면 `QUALITY_THRESHOLDS_INPUT_INVALID`로 fail-closed된다.
- local Quality Gate threshold criterion이 하나도 없으면 `QUALITY_THRESHOLDS_NOT_DECLARED`로 fail-closed된다.
- local Quality Gate threshold value가 numeric/finite/`[0.0, 1.0]` 조건을 만족하지 않으면 `QUALITY_THRESHOLD_VALUE_INVALID`로 fail-closed된다.
- local Quality Gate `thresholdProfile` classifies threshold intent. All-declared non-discriminating thresholds (`minimumTargetRecall<=0`, `minimumFindingPrecision<=0`, `maximumNegativeTargetFpr>=1`) produce `qualityGate.localQualityAssessment.status="not_decision_grade"` and final `qualityGate.status="not_decision_grade"` with `QUALITY_THRESHOLDS_NON_DISCRIMINATING`, even when split scoring runs. Missing metrics do not turn such runner-integrity profiles into quality failures; discriminating thresholds still fail missing metrics.
- offline Tool Portfolio `qualityDiagnostics` is diagnostic-only and primary-config scoped. It separates target score rows from unique raw finding pressure, buckets CWE by expected target CWE, exposes by-tool raw pressure without per-tool FP attribution, and must not change `qualityGate`/`decisionSupport`. `byCweTool` adds score-row-scoped expected-CWE/tool match-class buckets without per-tool FN/recall/precision/raw-pressure fields; missed targets remain portfolio-level in `byCwe`. Actual low-threshold diagnostic artifact: validation `uniqueRaw=226`, `tp=5`, `oracleFP=221`; test `uniqueRaw=311`, `tp=4`, `oracleFP=307`.

Stable order: `matching-policy-review`, `cwe-normalization-review`, `negative-discrimination-review`, `recall-gap-investigation`, `noise-pressure-review`. Raw-only `nonTpRawFindingCount` does not trigger noise triage without oracle-counted FP evidence.
- offline Tool Portfolio `toolContributionDiagnostics` is diagnostic-only current-six per-tool contribution/noise evidence. It uses aggregate all-scored-target metrics across `includedSplits`, derives only from single-tool and leave-one-out scores, emits stable `ALL_TOOLS` rows, and is `not_run` when comparative configs are incomplete. It never changes `qualityGate`/`decisionSupport` and never recommends add/remove/upgrade.
- offline Tool Portfolio report consumer canary exposes only sanitized neutral projections: gate statuses, threshold intent/status, diagnostic surface status, allowlisted reason codes/follow-ups, allowlisted diagnostic candidate IDs, and allowlisted current-six contribution classes. Unsafe scalar/string projection normalizes to `unknown`/`missing`, adds `TOOL_PORTFOLIO_REPORT_UNSAFE_PROJECTION`, and forces `toolPortfolioDecisionGradeUsable=false`.
- Tool Portfolio report consumer canary failure stderr uses exact structured diagnostics with `error`, singular `reasonCode`, legacy `reasonCodes`, `stage`, and `summaryEmitted=false`; successful summary stdout remains the stable consumer canary schema.
- The committed harness report artifact is locked against deterministic generator drift: `benchmark/results/tool_portfolio/s4-harness-fixture-report-v1.json` must exactly equal `build_harness_fixture_report(repo_root=services/sast-runner)` and include `qualityDiagnostics` + `toolContributionDiagnostics`. No other `benchmark/results/tool_portfolio/*.json` may use top-level schema `s4-tool-portfolio-experiment-report-v1`; stale same-schema artifacts are deleted rather than retained.
- local Quality Gate `primaryToolSetConfig`는 absent/null일 때만 `full-current-six`로 default되고, explicit blank/non-string/unknown 값은 `QUALITY_PRIMARY_TOOL_SET_CONFIG_INVALID`로 fail-closed된다.
- local Quality Gate diagnostic/snapshot output is sanitized independently of threshold evaluation: unknown `primaryToolSetConfig` values are reported as `<invalid>`, invalid primary-config snapshots are redacted even when another precondition failure wins, threshold snapshots emit only allowlisted fields, unknown threshold keys are omitted, invalid threshold values are `<invalid>`, invalid `requiredSplits` entries become `<invalid>` without raw split echo, and non-JSON threshold diagnostic paths redact unknown top-level/nested mapping keys.

Invalid split config must not populate `splitAssessments`, `passingSplits`, or `failingSplits` as if it were metric absence.
- Tool Portfolio report consumer canary explicitly allowlists and tests `QUALITY_REQUIRED_SPLITS_INVALID` as a safe non-decision-grade quality failure reason; it must not become `TOOL_PORTFOLIO_REPORT_UNSAFE_PROJECTION`.
- Caller-provided offline `systemStabilityGate` is now fully value-sanitized for non-pass evidence too: unknown top-level keys, invalid schema/reason codes, invalid requiredTools entries, and malformed phase/failure evidence fail closed without raw echo; `version` and `expectedExecutablePath` are redacted to status-only fields; non-pass minimal/subset/generated-empty requiredTools gates remain compatible, while pass gates still require complete current-six pass evidence.
- `corpusReadinessGate.status="blocked"`는 **decision-grade corpus readiness** 신호다. Caller-provided readiness gate 모순은 `CORPUS_READINESS_GATE_INCONSISTENT`로 blocked/not-decision-grade 정규화된다.
- `build_quality_gate()` is fail-closed even when called directly: quality eligibility requires explicit `systemStabilityGate.status="pass"` and `qualityGateAllowed is True`; absent system-gate evidence remains `SYSTEM_STABILITY_GATE_NOT_RUN`/`not_decision_grade`, valid `not_run` harness evidence is not blocked, and malformed or inconsistent direct-helper gates cannot become `eligible`.
- Direct `build_quality_gate()` reason propagation is allowlist-sanitized: system-stability reasons use the system top-level reason-code allowlist, corpus/readiness reasons use `READINESS_REASON_CODE_ALLOWLIST`, malformed entries add input-invalid sentinels, and arbitrary caller strings, numbers, and object `str`/`repr` values must not appear in `qualityGate.reasonCodes`.
- caller-provided `corpusReadinessGate`는 non-mapping/empty/invalid payload를 `CORPUS_READINESS_GATE_INPUT_INVALID`로 fail-closed한다. `available`은 required corpora, acquisitionStatuses, caseStatuses, summary checked/split counts, required-corpus-bound external projection을 모두 증명해야 하며, blocked/not_run은 minimal evidence를 보존하되 항상 top-level readiness blocker를 `requiredCorpusReadiness`로 projection한다.
- Corpus Readiness relative `localPath` with explicit base must stay inside that base after symlink resolution; `../` and symlink escapes block with `LOCAL_CORPUS_PATH_OUTSIDE_BASE` before case checks. Absolute corpus paths remain supported for `.omx` and externally pinned corpora.
- Offline experiment report final forbidden-key guard is value-sanitized: fixed forbidden verdict literals are canonicalized, arbitrary mapping labels are represented as `<field>`, and key objects are not stringified in diagnostics.
- Corpus manifest public identifiers are value-sanitized before scoring: unsafe `caseId`, `expected.targetId`, explicit `lineageId`, or external `acquisitionId` fail closed with field-only diagnostics; duplicate case IDs and cross-split lineage/source-artifact leakage do not echo caller-controlled IDs; public `lineageId` fallback is safe `caseId`, not `sourceRef`.

Readiness preserves deterministic blocked JSON for those unsafe paths with `CORPUS_CASE_SOURCE_PATH_UNSAFE`, including literal backslash filenames that exist and checksum-match on Linux, and redacts unsafe status paths as `sourcePath="<unsafe>"`/`sourcePathStatus="unsafe"`; CLI invalid-output JSON redacts raw exception messages/paths with fixed `error="input validation failed"` plus `errorClass`, output-write failure uses `CORPUS_READINESS_OUTPUT_WRITE_FAILED` via stdout fallback, and readiness statuses replace host-local paths with frozen enum fields (`resolvedPathStatus`, `localPathStatus`, `resolvedLocalPathStatus`) instead of raw paths. Current allowed vocabularies: `localPathStatus=available|base_required|missing|not_declared|not_resolved|outside_base`, `resolvedLocalPathStatus=available|missing|not_resolved|outside_base`, `resolvedPathStatus=available|checksum_mismatch|missing|outside_root|unsafe`, `sourcePathStatus=unsafe`. Report-side caller-provided `corpusReadinessGate` payloads are validated against this same path/status contract: available gates require available status proof, non-allowlisted values and contradictory available+unsafe status fail closed, raw host path fields are forbidden even on extra acquisition entries, and unsafe `caseStatuses[].sourcePath` values are rejected without raw echo. Caller-provided top-level `systemStabilityGate.status` and `corpusReadinessGate.status` input diagnostics do not echo invalid arbitrary strings or stringify non-string objects; they use `<invalid>`/`<blank>` or safe type-only diagnostics. Nested `systemStabilityGate` pass-evidence diagnostics also avoid caller-value echo: invalid unknown `requiredTools[]` values and invalid phase status strings are redacted, non-string values report safe type-only fields, and generated missing-current-six tool diagnostics remain unchanged. Invalid legacy `external_corpus_status` diagnostics likewise avoid arbitrary key/field echo or object stringification; accepted valid legacy compatibility context remains visible but invalid diagnostic labels use `<invalid>` plus safe type metadata. Invalid `matchingPolicy` diagnostics also avoid raw echo/stringification for unknown keys, schemaVersion values, and out-of-range lineWindow values; diagnostics keep deterministic category/field/type/expected evidence. Actual-runner direct staging rejects relative acquisition `localPath` without explicit base instead of resolving against cwd; report mode remains blocked/no-scan through `LOCAL_CORPUS_BASE_PATH_REQUIRED`.
- Required corpus IDs are strict safe strings; invalid non-string/whitespace/control/oversized IDs produce `CORPUS_REQUIRED_CORPUS_ID_INVALID` without raw echo. Safe unknown IDs remain valid required corpora but project as generic `external` rather than raw external status keys.
- Caller-provided readiness external projections are sanitized before report output: only canonical readiness keys, allowed nested fields, validated acquisition IDs, and allowlisted readiness reason codes survive; safe-shaped malicious keys/reasons/nested fields are dropped or input-invalid and do not reach `corpusReadinessGate`/`decisionSupport`.
- legacy `external_corpus_status`는 compatibility context로 `decisionSupport.externalCorpusStatus`에 남을 수 있지만 final `qualityGate`와 `requiredFollowUps`를 결정하지 않는다. 그 둘은 readiness-derived projection만 소비한다.
- 세 Gate는 독립이다. 도구가 모두 살아 있어도 local corpus가 없거나 validation/test 품질 threshold가 실패할 수 있다.

### 기동 / 환경

```bash
# 사용자 허락 없이 실행하지 말 것
./scripts/start-sast-runner.sh
tail -20 logs/s4-sast-runner.jsonl
```

- `scripts/start-sast-runner.sh`는 기본적으로 `uvicorn --reload --reload-dir app`로 실행된다.
- 핫 리로드를 끄고 단일 프로세스로 띄우려면 `SAST_HOT_RELOAD=0 ./scripts/start-sast-runner.sh`를 사용한다.

`.env` / 기본값 예시:
```env
SAST_PORT=9000
SAST_SCAN_TIMEOUT=120
SAST_MAX_CONCURRENT_SCANS=2
SAST_SDK_ROOT=/opt/sdks   # 예시값 — 환경별로 교체
```

**주의**: `list[str]` 타입 필드를 `.env`에 쓰면 pydantic-settings JSON 파싱 실패. `str` 타입 + `@property`로 우회 (config.py 참조).

## 3. 핵심 설계 원칙

- **결정론적 처리 최대화, LLM 결정 표면 최소화**
- **도구별 profile 분리** — 컴파일 기반 도구만 SDK enriched, 나머지는 original
- **scope-early** — `thirdPartyPaths` 파일을 도구 실행 전에 제외 (OOM 방지)
- **경계면 분석** — SDK/라이브러리 경로 finding이라도 dataFlow에 사용자 코드 포함 시 유지 (`origin: "cross-boundary"`)
- **gcc-fanalyzer는 `-c`** (`-fsyntax-only`에서는 analyzer가 실행 안 됨)
- **scan-build는 `-plist`** (없으면 plist 파일 미생성)
- **파일별 개별 실행** (gcc-fanalyzer, scan-build — 동일 심볼 충돌 방지)
- **CWE는 전 도구에서 태깅** — scan-build/gcc-fanalyzer도 매핑 추가 완료
- **Semgrep taint + sanitizer** — source/sink 자동 추적 + 가드 패턴 제외

## 4. 내부 SDK 해석 데이터 (analysis path only)

```text
$SAST_SDK_ROOT/              <- .env 예시: SAST_SDK_ROOT=/opt/sdks (환경별로 교체)
  ├── sdk-registry.json       <- SDK 메타데이터 (외부 설정, 코드 밖)
  └── ti-am335x/              <- sdkId = 폴더명
```

| sdkId | SDK | GCC 버전 | 비고 |
|-------|-----|:---:|------|
| `ti-am335x` | TI AM335x 08.02.00.24 | 9.2.1 | `-fanalyzer` 미지원 -> 호스트 gcc 폴백 또는 SDK gcc 재확인 |

**중요**: `/v1/sdk-registry` public API는 제거되었다.
이 데이터는 현재 analysis path 내부 해석에만 남아 있으며, build path는 더 이상 `sdkId`를 받지 않는다.

정식 S1/S2 SDK upload/registration 루트 밖에서 S3가 서비스를 단독 테스트할 때는 임의 `sdkId`나 `custom` sentinel을 보내지 말고, `sdkResolutionMode="non-registered"` + `sdkDescriptor.sdkRootPath`를 보내야 한다. S4는 이 descriptor에서 include/sysroot/compiler/setup 후보를 결정론적으로 산출하며 registry를 조회하지 않는다. SDK가 없음이 명확한 경우는 `sdkResolutionMode="none"`을 사용한다.
